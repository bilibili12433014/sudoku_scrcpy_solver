# 数独助手

这是一个基于 `Python + tkinter + OpenCV + pywin32 + adb + scrcpy` 的桌面工具，用来读取 Android 设备上的数独界面、计算当前可直接确定的一步，并通过 ADB 回填到手机。

项目目标不是“暴力解出所有题”，而是：

- 稳定读取当前手机上的数独盘面
- 只给出确定步，不猜测
- 尽量避免误识别和误输入
- 在出现问题时留下足够日志，方便复盘和修 bug

## 当前功能

- 自动启动 `scrcpy`
- 自动解析 `scrcpy` 输出中的设备序列号、窗口标题、手机分辨率
- 使用 `PrintWindow` 对 `scrcpy` 窗口做离屏截图
- 自动裁掉标题栏、窗口边框和黑边
- 自动识别数独棋盘和底部 1-9 输入区域
- 自动识别题面已有数字、已填正确数字、选中态和同号高亮态
- 计算下一步可直接确定的数字
- 在界面中显示当前棋盘和建议步
- 通过 ADB 点击棋盘格和底部数字完成回填
- 支持单步“操作”
- 支持“自动操作”，连续执行直到填满或没有更多确定步
- 将关键流程、状态、异常和棋盘快照写入 `./log.log`

## 文件说明

- `sudoku_scrcpy_solver.py`
  主程序，包含 GUI、窗口管理、图像识别、求解逻辑、ADB 回填和日志逻辑。
- `log.log`
  运行时日志文件。程序启动后会自动创建或追加写入。
- `temp.bmp`
  调试阶段保留的截图样本，不参与主流程。

## 运行环境

需要满足以下条件：

- Windows
- Python 3
- 系统 PATH 中可直接使用 `scrcpy` 和 `adb`
- 已安装 Python 包：
  - `opencv-python`
  - `numpy`
  - `pywin32`
- `tkinter` 可用

如果 Python 环境缺包，可以直接安装：

```powershell
pip install opencv-python numpy pywin32
```

## 启动方式

在项目目录中运行：

```powershell
python sudoku_scrcpy_solver.py
```

程序会自动启动 `scrcpy`，不需要手动先开窗口。

## 使用流程

### 1. 读取当前画面

- 勾选“读取”会持续轮询当前 `scrcpy` 画面。
- 如果不想自动轮询，可以取消勾选。
- 初次使用时建议先勾选“读取”，确认棋盘识别正确。

### 2. 计算下一步

- 点击“下一步”后，程序会根据当前棋盘只计算一个“确定步”。
- 当前实现的求解策略包含：
  - 唯一候选
  - 行唯一
  - 列唯一
  - 宫唯一
  - 裸对
  - 裸三元组
  - 隐藏对
  - 宫区块摒除
  - 区块摒除
  - X-Wing
  - XY-Wing
- 候选消减类技巧不会直接“猜一个数”，而是先删掉不可能候选，再立刻检查是否出现新的确定步。
- 如果当前没有可直接确定的一步，会提示停止，不会猜。

### 3. 执行一步

- 点击“操作”时：
  - 如果当前还没有“下一步”，程序会先自动计算一步
  - 然后检查目标格当前是否仍为空
  - 通过 ADB 点击目标格
  - 再点击底部数字
  - 再读取目标格确认是否成功

### 4. 连续执行

- 点击“自动操作”后，程序会循环执行：
  - 计算下一步
  - 回填
  - 再计算下一步
- 直到：
  - 棋盘填满
  - 当前局面没有更多可直接确定的步骤
  - 中途某一步确认失败

## 界面说明

### 操作区

  控制是否自动同步当前画面。
- `下一步`
  计算一手可直接确定的步骤，并在棋盘中高亮。
- `操作`
  执行当前步骤；如果当前没有步骤，会先自动计算。
- `自动操作`
  连续执行直到停止条件触发。

  - 已填并确认的数字：`#CB7118`
  - 建议填写的当前格：蓝色显示建议值
  - 当前建议格所在行、列、宫：`#FEECCC`

### 信息区

- 显示已识别数字总数
- 显示已填并确认的格子数量

- 显示最近一次成功执行的结果

## 总体实现结构

程序主要由四部分组成。

### 1. `ScrcpyManager`

负责：

- 启动和关闭 `scrcpy`
- 解析 `scrcpy` 控制台输出
- 获取设备序列号、窗口标题、手机分辨率
- 定位窗口句柄
- 通过 `PrintWindow` 对窗口做离屏截图
- 使用 `adb shell input tap` 执行点击

设计要点：

- 不使用普通屏幕截图，避免窗口被遮挡时截到遮挡层
- 如果窗口最小化，则直接报错
- 如果截图接近全黑，则认为离屏截图失败

### 2. `SudokuVision`

负责：

- 识别棋盘位置
- 识别底部数字输入区位置
- 从底部 1-9 生成数字模板
- 识别整个棋盘
- 单独识别某一个格子

设计要点：

- 棋盘定位基于颜色 `#D78826` 的近似 HSV 范围
- 读取时会同时缓存：
  - 当前棋盘
  - 已填正确数字位置
  - 当前数字模板
  - 棋盘几何信息
- 模板来自当前界面底部数字条，而不是外部 OCR 或预训练模型

### 3. `SudokuSolver`

负责纯逻辑求解。

当前实现的确定性技巧包括：

- 唯一候选
- 行隐藏单数
- 列隐藏单数
- 宫隐藏单数
- 裸对
- 裸三元组
- 隐藏对
- 宫区块摒除
- 区块摒除
- X-Wing
- XY-Wing

它不会：

- 猜测
- 回溯
- 暴力搜索

求解顺序是：

1. 先直接找可以落子的格子
2. 如果没有，就做一轮候选消减
3. 每做完一轮候选消减，立刻重新检查是否出现新的唯一候选或隐藏单数
4. 如果仍然没有确定步，就停止

这符合“最多错 3 次，不要猜”的要求。

### 4. `SudokuApp`

负责：

- tkinter GUI
- 用户按钮事件
- 自动读取轮询
- 自动操作循环
- 棋盘显示
- 状态提示
- 日志记录

## 求解算法与步骤

### 已实现算法

#### 1. 唯一候选

某个空格的候选集只剩 1 个数时，直接填入。

例如：

- 某格当前候选为 `{6}`，则该格必填 `6`

#### 2. 行唯一

某个数字在一整行的所有空格候选中只出现一次，即使该格还不止一个候选，也可以确定填入。

#### 3. 列唯一

某个数字在一整列的所有空格候选中只出现一次，可以直接填入。

#### 4. 宫唯一

某个数字在一个 3x3 宫的所有空格候选中只出现一次，可以直接填入。

#### 5. 裸对

如果一个单位内有两个空格的候选完全相同，并且都只包含同两个数字，例如 `{2, 9}`、`{2, 9}`，那么这两个数字不可能出现在该单位的其他空格里。

程序会：

1. 在行、列、宫三个维度搜索裸对
2. 从同单位其他格删掉这两个候选
3. 立即重新检查是否出现新的唯一候选或隐藏单数

#### 6. 宫区块摒除

如果某个 3x3 宫里，某个数字的所有候选都落在同一行或同一列，那么这个数字在该行或该列的宫外位置都不可能再出现。

这对应常见的 `Pointing Pair / Pointing Triple`。

程序会：

1. 在每个宫内统计每个数字的候选位置
2. 如果这些位置全部落在同一行，就从这行的宫外空格删掉该数字
3. 如果这些位置全部落在同一列，就从这列的宫外空格删掉该数字
4. 删完后立刻重新检查是否出现确定步

#### 7. 裸三元组

如果一个单位内存在三个空格，它们的候选合集刚好只有三个数字，例如：

- `{1, 2}`
- `{1, 3}`
- `{2, 3}`

或者：

- `{2, 5}`
- `{2, 5, 8}`
- `{5, 8}`

那么这三个数字只能落在这三个格里，不能再出现在该单位的其他空格中。

程序会：

1. 在行、列、宫三个维度枚举候选长度为 2 或 3 的三格组合
2. 只在“三格候选合集大小正好为 3，且这三个数字只被这三格覆盖”时生效
3. 从同单位其他格删除这三个数字
4. 删完后立刻重新检查是否出现确定步

#### 8. 隐藏对

如果某个单位内有两个数字，它们都只出现在同样的两个格子里，那么这两个格子就只能保留这两个数字，其他候选都可以删掉。

例如数字 `4` 和 `9` 都只出现在同一行的 `c2`、`c7` 两格中，那么这两格都可以压缩成 `{4, 9}`。

程序会：

1. 在行、列、宫三个维度统计每个数字出现在哪些候选格中
2. 找出“两个数字都只落在同样两格”的情况
3. 把这两格的候选裁剪为这两个数字
4. 裁剪后立刻重新检查是否出现确定步

#### 9. 区块摒除

如果某一行或某一列里，某个数字的所有候选都落在同一个 3x3 宫，那么这个数字在该宫的其他格里都不可能再出现。

这对应常见的 `Claiming Pair / Claiming Triple` 或 `Box-Line Reduction`。

程序会：

1. 逐行检查每个数字的候选位置是否锁定在同一宫
2. 逐列做同样检查
3. 从该宫的其他格删除这个数字
4. 删完后立刻重新检查是否出现确定步

#### 10. X-Wing

如果某个数字在两行中都只出现在同样两列，或者在两列中都只出现在同样两行，就形成一个矩形锁定。

这时：

- 这两列的其他行不能再保留该数字
- 或这两行的其他列不能再保留该数字

程序当前实现的是标准二维 `X-Wing`：

1. 逐个数字检查所有行，找出“恰好两列”且列坐标完全一致的两行
2. 从其他行的这两列中删除该数字
3. 再逐个数字检查所有列，找出“恰好两行”且行坐标完全一致的两列
4. 从其他列的这两行中删除该数字
5. 每次删完都立即重新检查是否出现确定步

#### 11. XY-Wing

`XY-Wing` 由 3 个双值格组成：

- 一个枢纽格，候选为 `{X, Y}`
- 一个翼格，候选为 `{X, Z}`
- 另一个翼格，候选为 `{Y, Z}`

两个翼格都必须和枢纽格同处某个单位，但不要求彼此同行、同列或同宫。这样一来，两个翼格公共可见区域里的 `Z` 就可以被删除。

程序会：

1. 遍历所有候选数长度为 2 的格子，寻找可作为枢纽的 `{X, Y}`
2. 在与枢纽同单位的双值格里寻找 `{X, Z}` 和 `{Y, Z}`
3. 找到后，从同时能看见这两个翼格的其他格中删掉 `Z`
4. 删完后立即重新检查是否出现确定步

### 算法执行顺序

每次点击“下一步”或自动求解时，求解器按下面的顺序工作：

1. 唯一候选
2. 行唯一
3. 列唯一
4. 宫唯一
5. 裸对
6. 裸三元组
7. 隐藏对
8. 宫区块摒除
9. 区块摒除
10. X-Wing
11. XY-Wing
12. 每完成一次候选消减，重新回到第 1 步

如果没有任何一步能推出确定数字，就返回“当前局面没有找到可直接确定的一步”。

### 安全边界

- 所有高级技巧都只是删除不可能候选，不会猜数
- 一旦候选消减导致某个空格候选数被删空，程序会直接报错并停止
- 不会使用回溯或暴力搜索去填数

## 图像识别技术细节

### 1. 棋盘定位

棋盘边框颜色固定为橙色，因此先转 HSV，然后提取接近该橙色的区域，再找接近正方形的大轮廓作为候选棋盘。

之后会在棋盘区域内继续取竖线和横线投影，估计 10 条网格线。如果线提取不稳定，会退回到等分几何估计。

### 2. 输入区定位

底部数字条颜色会在棕色和灰色之间变化，所以会同时提取：

- 启用数字的棕色范围
- 禁用数字的灰色范围

最后合并掩码，定位整条输入区。

### 3. 数字模板生成

底部 1-9 会被裁成 9 个区域，每个区域提取数字掩码后归一化，作为模板。

这样有两个好处：

- 不依赖外部 OCR
- 与当前界面的字体、缩放、抗锯齿状态天然一致

### 4. 多状态数字识别

数独格子存在多种视觉状态：

- 浅底 + 棕字
- 浅底 + 橙字
- 深棕底 + 白字
- 绿色高亮底 + 白字

单一阈值很容易误判，所以当前实现会为每个格子生成多组候选掩码：

- 基于与背景色差的掩码
- 基于灰度明暗差的掩码
- 基于白字条件的掩码
- 基于自适应阈值的掩码

每个掩码再提取若干个候选连通域，和当前模板逐一匹配，选出分数最高的一组。

### 5. 已填正确数字识别

识别到数字后，会用数字像素的平均颜色与：

- 原始数字棕色 `#9C612F`
- 已填正确橙色 `#CB7118`

做距离比较，以区分“题目原始数字”和“程序或用户后续填入的正确数字”。

## 回填技术细节

### 1. 坐标换算

截图是在 `scrcpy` 窗口内容区域上完成的，但点击需要发送到手机真实分辨率。

因此会把：

- 棋盘格中心
- 底部数字按钮中心

从内容区域坐标按比例映射到真实设备分辨率，再发给 ADB。

### 2. 操作前检查

执行“操作”前不会重新强制识别整盘，而是只检查目标格当前是否仍为空。

这样做的原因是：

- 整盘重识别更容易被个别高亮格干扰
- 目标格校验更稳定
- 用户手动改动后也能尽快发现并同步

### 3. 操作后确认

点击后不会依赖底部数字条的实时状态重新生成模板，而是复用上一次稳定读取时缓存的模板，只对目标格做确认。

确认过程会做最多两次轻量重试。

## 自动操作细节

自动操作启动后会暂时禁用：

- 读取
- 下一步
- 操作
- 自动操作

同时自动读取轮询会暂停，避免和自动回填冲突。

循环逻辑为：

1. 判断棋盘是否已满
2. 如果没有当前步骤，则自动求一步
3. 执行当前步骤
4. 成功后继续下一轮
5. 失败或无确定步时停止

## 常见问题

### 1. 窗口截图失败

可能原因：

- `scrcpy` 尚未完全启动
- `scrcpy` 窗口被最小化
- `PrintWindow` 返回了异常结果

建议：

- 等待几秒后重新读取
- 不要最小化 `scrcpy`

### 2. 点击后明明成功了，但程序提示未确认

这通常与目标格的视觉状态变化有关，例如：

- 深色选中态
- 绿色同号高亮态
- 窗口缩放后局部几何发生变化

这类问题请优先提供：

- 触发问题时的截图
- 对应时段的 `log.log`

### 3. 自动操作提前停止

可能原因：

- 当前已经没有“确定步”
- 某一步回填后确认失败
- 识别到目标格状态变化

程序会尽量停在安全状态，而不是冒险继续。

## 已知限制

- 当前求解策略仍然只做“确定性”推理，不包含回溯、猜测或暴力搜索
- 目前已实现到 XY-Wing；更高阶技巧如隐藏三元组、Swordfish、XYZ-Wing 还未加入
- 完全依赖当前游戏 UI 的颜色和布局特征
- 如果游戏大幅改版，识别阈值可能需要重新调
- 如果窗口被最小化，离屏截图不可用
- 多设备同时连接时，程序依赖 `scrcpy` 输出中解析到的当前序列号

## 后续可扩展方向

- 增加“停止自动操作”按钮
- 增加更高级数独技巧，例如隐藏三元组、Swordfish、XYZ-Wing
- 增加调试导出：自动保存失败帧
- 增加模板和识别中间结果可视化
- 增加日志分级和按日期滚动

## 维护建议

如果后续需要排查问题，建议按这个顺序：

1. 查看 `log.log`
2. 对照出错时的截图
3. 看目标格是在什么视觉状态下失败
4. 再决定是调：
   - 棋盘定位
   - 模板提取
   - 单格识别
   - 操作后确认逻辑

如果是继续迭代本项目，优先保持以下原则：

- 不猜
- 先确认再回填
- 出错时保留足够上下文
- 让日志能独立复盘一次完整流程
